# Common False Positives in AI-Generated SEO Audits

A catalog of specific claims AI audits routinely make that are usually wrong, plus the rebuttal pattern for each. Refer to this list whenever you're verifying another AI's audit or running your own — match incoming claims against the patterns here before accepting them.

## "No robots.txt found"

**Why it's usually wrong:** The auditor probably didn't actually fetch `/robots.txt`. Most static-site frameworks (Hugo, Jekyll, Next.js, custom build scripts) generate one by default.

**Verify:** `curl -s https://SITE.com/robots.txt | head`

**Rebuttal:** "robots.txt exists at /robots.txt, content: [first lines]. Sitemap reference present: yes/no."

## "Missing XML sitemap"

**Why it's usually wrong:** Same root cause — auditor didn't curl `/sitemap.xml` or check `robots.txt` for the Sitemap directive.

**Verify:** `curl -s https://SITE.com/sitemap.xml | head -5; wc -l < <(curl -s https://SITE.com/sitemap.xml)`

**Rebuttal:** "Sitemap exists at /sitemap.xml with N URLs."

## "Missing canonical tags"

**Why it's usually wrong:** Canonical tags live in `<head>`, and AI auditors often inspect rendered visible content rather than view-source.

**Verify:** `curl -s https://SITE.com/PAGE/ | grep -oE 'rel="canonical"[^>]*'`

**Rebuttal:** "Canonical present on /PAGE/, points to [URL]."

## "No JSON-LD / structured data detected"

**Why it's usually wrong:** Same as above — `<script type="application/ld+json">` blocks live in head or just before `</body>`.

**Verify:** `curl -s https://SITE.com/PAGE/ | grep -c 'application/ld+json'` then enumerate types.

**Rebuttal:** "Page contains N JSON-LD blocks of types [list]." If gaps exist, name them specifically rather than claiming "missing."

## "Missing meta descriptions"

**Why it's usually wrong:** Most CMS / build scripts have a default meta description generator; absent ones are usually intentional (e.g., 404 pages, taxonomy pages).

**Verify:** `curl -s https://SITE.com/PAGE/ | grep -oE '<meta name="description"[^>]+content="[^"]*"'`

**Rebuttal:** Cite the actual meta description or confirm absence.

## "Missing Article schema on guide/blog pages"

**Why it's usually wrong:** Article schema is auto-generated by most blogging platforms. The auditor may have looked at non-article pages and generalized.

**Verify:** Check a representative article URL with grep.

## "Wrap decorative icons in role='img' aria-label='...'"

**Why it's WRONG:** This advice is backwards for decorative icons. WCAG 1.1.1 distinguishes decorative from informative imagery. Decorative icons (where adjacent text conveys the meaning) should be HIDDEN from assistive tech, not labeled. Labeling them creates announcement noise.

**The test:** Would a screen-reader user lose meaning if the icon were removed? If no → decorative → `aria-hidden="true"`. If yes → informative → `aria-label` or `alt`.

**Rebuttal:** "Decorative icons should use `aria-hidden='true'`, not `role='img' aria-label='...'`. The adjacent H3 already conveys the meaning; labeling the icon would add noise. (Reference: WCAG 1.1.1 Non-text Content.)"

## "Add FAQ schema for rich snippets"

**Why it's usually STALE:** Google deprecated FAQ rich results for most sites in May 2026 — they no longer appear in search results except for authoritative government/health sites. The structured data is still useful for entity understanding, but the prominent SERP-treatment benefit is gone.

**Verify:** Search current Google Search Central docs for "FAQPage structured data" — note the deprecation status.

**Rebuttal:** "FAQ rich results were deprecated in May 2026 for most sites; the structured data still aids entity understanding but no longer produces visible SERP rich results. Worth adding only if optimizing for entity recognition, not for SERP real estate."

## "Add AMP"

**Why it's WRONG:** AMP was deprecated as a Top Stories requirement in 2021 and broadly de-emphasized after. Modern sites don't need it.

**Rebuttal:** "AMP is no longer a ranking signal or a Top Stories requirement. Focus on standard Core Web Vitals instead."

## "Add hreflang"

**Why it's USUALLY irrelevant:** Only relevant if the site has multi-language or multi-region versions. Single-language sites should not add hreflang.

**Rebuttal:** "Site is single-language; hreflang would be incorrect to add. If multi-language is on the roadmap, revisit at that point."

## "Use SRCSET on every image" / "Convert all images to WebP"

**Why it's nuanced:** Real wins for hero images and infographics; marginal for icons and small decorative images. Blanket recommendations ignore the build-time cost.

**Verify:** Run PageSpeed Insights — if LCP is image-bound, focus there. If LCP is text or first-contentful-paint, image optimization is a smaller lever.

**Rebuttal:** Recommend only for the specific images identified as LCP bottlenecks or that are large enough to matter (>50KB).

## "Variant pages cause keyword cannibalization, canonicalize them all to parent"

**Why it's a DECISION, not a fix:** Programmatic SEO sites intentionally generate variant pages to catch long-tail queries. The auditor's "consolidate to parent" advice kills that strategy. Both consolidation and self-canonical are defensible — depends on whether variants actually rank.

**Rebuttal:** "Variant pages with unique H1/title targeting distinct queries are a valid pSEO pattern. Consolidating sacrifices long-tail capture for signal concentration. Check Search Console for variant-page impressions before deciding — if they're earning clicks, keep self-canonical; if they're being ignored or chosen-against by Google, consider consolidation."

## "Site is missing X-Content-Type-Options / HSTS / X header"

**Verify:** `curl -sI https://SITE.com/` and check headers directly. Many static-hosting providers (Cloudflare, Netlify, Vercel) auto-add these or they're in the project's `_headers` file.

## Pattern: "the auditor couldn't read the head"

The unifying false-positive pattern is: AI auditors that produce summaries from rendered visible content (post-CSS) cannot see what's in `<head>`. So they confabulate absences for everything that lives in head — meta tags, structured data, link rels, preload hints. Always treat "missing X" claims about head-content as needing curl/grep verification.

When you encounter a new "missing X" claim not on this list, ask: "Is X something that would live in head, _headers, robots.txt, sitemap, or a generated file?" If yes, the claim needs verification before acceptance.

## Pattern: "the advice is from 2-3 years ago"

The other unifying pattern is stale advice. Whenever a recommendation involves:
- Specific structured data types
- Rich-result eligibility
- A "Page Experience" metric or weight
- A Google update name (Helpful Content, Core Update, etc.)
- An algorithm signal (E-E-A-T variants, etc.)

…assume the recommendation may be outdated and web-search the current Google Search Central docs to verify before reporting.
